# VJLive3 Swarm Runbook

This document contains instructions for operating and debugging the Swarm infrastructure (specifically the Orange Pi RK3588 NPUs running the Qwen 4B model for Phase 1 Skeleton Generation).

## 1. Boot Sequence & Initialization
If `batch_specs.py` fails to connect (`<urlopen error [Errno 111] Connection refused>`), the API daemon `spec_server.py` is not running on the Orange Pis.

To manually initialize the NPUs:
1. SSH into the worker node(s):
   - Julie: `ssh happy@192.168.1.60`
   - Maxx: `ssh happy@192.168.1.50`
2. **STRICT ISOLATION PROTOCOL:** Do NOT run the script from the legacy `/home/happy/outback` directory to prevent cross-contamination.
3. Traverse into the mounted Desktop workspace:
   ```bash
   cd "/home/happy/Desktop/claude projects/VJLive3_The_Reckoning/agent-heartbeat/"
   ```
4. Start the API Service:
   ```bash
   python3 spec_server.py &
   ```
5. *Wait.* The RKLLM toolkit will allocate the NPU cores and load the `w8a8` quantized weights into memory. The process will bind to port `5050` when the model is ready to serve endpoints.

## 2. Execution & Benchmarks
From your Desktop Host, trigger the generation pipeline:
```bash
python3 agent-heartbeat/batch_specs.py
```

### ⏱️ Performance Benchmarks (DO NOT ABORT EARLY)
The RKLLM generation is **NOT** instantaneous. The process is computationally heavy for the NPU.
- **Expected Time per Spec:** ~5 to 6 minutes (300 - 360 seconds).
- **Behavior:** The script will appear to "hang" while waiting for the NPU to complete the sequence generation.
- **Monitoring:** Do not assume it is broken. Monitor the logs directly to see completion tokens:
  ```bash
  tail -f agent-heartbeat/logs/batch_specs.log
  ```
  *You will see outputs like: `✅ P4-BA01: 344s` indicating successful generation.*

## 3. Directory Routing (The Queue)
Once tasks are generated by the NPUs, the `swarm_sync.sh` script or the `batch_specs` controller deposits the completed raw outputs safely into the routing queue:
`docs/specs/_01_skeletons/`

From there, Desktop Roo instances or Worker Roo instances claim the skeletons by strictly moving them into `_02_active_...` for processing into `_02_fleshed_out/`.
